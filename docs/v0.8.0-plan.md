请基于当前 `JustinQP/codex-job` 项目，完成 `v0.8.0：App Server 会话模式主线集成`。

## 背景

当前项目已有两条链路：

1. 主线任务链路：

```text
FastAPI Backend -> SQLite/SQLModel -> HTTP Runner -> codex exec
```

2. App Server POC 链路：

```text
poc/app_server -> HTTP Bridge -> codex app-server -> mobile.html
```

v0.7.1 已完成：

* App Server POC parser 测试
* App Server Bridge API 测试
* rerun_task 参数保留
* 主线 mobile XSS escape
* README 中主线 mobile 与 App Server POC mobile 边界说明

现在进入 v0.8.0：把 App Server 会话能力以 sidecar 方式接入主系统。

重要原则：

```text
不要把 poc/app_server/app_server_bridge.py 合并进 backend/main.py。
不要替换现有 Runner/codex exec 主链路。
App Server Bridge 继续作为 sidecar。
主后端只通过 HTTP 调用 Bridge。
```

目标架构：

```text
手机 / Web / API
  ↓
backend/main.py
  ↓
App Thread API
  ↓
backend/services/app_server_bridge_client.py
  ↓ HTTP
poc/app_server/app_server_bridge.py
  ↓ stdio
codex app-server
```

# 四、Bridge Client

新增文件：

```text
backend/services/app_server_bridge_client.py
```

职责：

```text
主后端通过 HTTP 调用 App Server Bridge sidecar。
```

使用 Python 标准库，优先 `urllib.request`，不要新增 requests/httpx 依赖。

## 环境变量

支持：

```text
APP_SERVER_BRIDGE_URL
APP_SERVER_BRIDGE_TOKEN
APP_SERVER_BRIDGE_TIMEOUT_SECONDS
```

默认：

```text
APP_SERVER_BRIDGE_URL=http://127.0.0.1:8766
APP_SERVER_BRIDGE_TIMEOUT_SECONDS=300
```

## Client 方法

实现类或函数均可，但要清晰。

至少提供：

```text
get_health()
create_thread(title: str)
list_threads()
get_thread(bridge_thread_id: str)
rename_thread(bridge_thread_id: str, title: str)
delete_thread(bridge_thread_id: str)
send_turn(bridge_thread_id: str, message: str)
get_final(bridge_thread_id: str)
get_events(bridge_thread_id: str)
```

## 错误处理

定义自定义异常：

```text
AppServerBridgeError
```

异常信息至少包含：

```text
status_code
code
message
step
```

要求：

1. HTTP 4xx/5xx 转成 AppServerBridgeError。
2. 网络连接失败转成 AppServerBridgeError。
3. JSON 解析失败转成 AppServerBridgeError。
4. timeout 转成 AppServerBridgeError。
5. 不泄露 token。

# 五、后端 Service

新增文件：

```text
backend/services/app_thread_service.py
```

职责：

```text
管理 AppThread/AppTurn 持久化，并调用 app_server_bridge_client。
```

## 函数建议

实现以下函数：

```text
create_app_thread(session, project_id, title)
list_app_threads(session, project_id=None, limit=50)
get_app_thread_or_404(session, app_thread_id)
rename_app_thread(session, app_thread_id, title)
close_app_thread(session, app_thread_id)
send_app_turn(session, app_thread_id, message)
list_app_turns(session, app_thread_id)
get_app_thread_final(session, app_thread_id)
get_app_thread_events(session, app_thread_id)
```

## 行为要求

### 1. create_app_thread

流程：

```text
校验 project 存在
校验 project.enabled = true
调用 Bridge POST /threads 创建 bridge thread
保存 AppThread
返回 AppThreadRead
```

如果 Bridge 不可用：

```text
不创建 AppThread
返回 502 或 503
```

### 2. send_app_turn

流程：

```text
校验 AppThread 存在
校验 AppThread.status 不是 CLOSED
创建 AppTurn(status=RUNNING)
调用 Bridge POST /threads/{bridge_thread_id}/turns
保存 assistant_final / bridge_turn_id / event_summary_json
AppTurn.status=SUCCESS
AppThread.status=ACTIVE
AppThread.updated_at=now
```

失败时：

```text
AppTurn.status=FAILED
AppTurn.error_message=...
AppThread.status=ERROR
AppThread.last_error=...
返回 502/504/500 中合适状态
```

### 3. close_app_thread

流程：

```text
如果有 bridge_thread_id，调用 Bridge DELETE /threads/{id}
即使 Bridge 已经 404，也允许本地标记 CLOSED
AppThread.status=CLOSED
```

### 4. rename_app_thread

流程：

```text
更新本地 title
如果存在 bridge_thread_id，尽量调用 Bridge PATCH 同步标题
Bridge 失败时返回错误，不静默吞掉
```

### 5. list_app_turns

按 id 正序返回。

### 6. event_summary_json

第一版不要持久化全部 events，只保存 Bridge 返回的 summary JSON。

# 七、API

修改：

```text
backend/main.py
```

新增 API：

```text
GET    /app-threads
POST   /app-threads
GET    /app-threads/{app_thread_id}
PATCH  /app-threads/{app_thread_id}
DELETE /app-threads/{app_thread_id}

POST   /app-threads/{app_thread_id}/turns
GET    /app-threads/{app_thread_id}/turns
GET    /app-threads/{app_thread_id}/final
GET    /app-threads/{app_thread_id}/events
GET    /app-server-bridge/health
```

全部接口除 `/health` 外，继续使用现有 `require_api_token`。

## 1. GET /app-server-bridge/health

调用 Bridge `/health`，返回 Bridge 状态。

如果 Bridge 不可用：

```text
返回 503
{
  "status": "unavailable",
  "error": "..."
}
```

## 2. POST /app-threads

请求：

```json
{
  "project_id": 1,
  "title": "xxx"
}
```

行为：

```text
创建主系统 AppThread，并创建 Bridge thread。
```

## 3. POST /app-threads/{id}/turns

请求：

```json
{
  "message": "..."
}
```

行为：

```text
同步调用 Bridge send turn。
第一版允许阻塞等待完成。
```

如果 Bridge timeout：

```text
AppTurn 标记 FAILED
返回 504
```

## 4. DELETE /app-threads/{id}

行为：

```text
关闭 Bridge thread
本地标记 CLOSED
```

不用物理删除数据库记录。

# 八、主线 mobile 页面增强

修改：

```text
backend/mobile.py
```

目标：

在现有主线手机控制台中新增一个 “App Server 会话” 区块。

不要重写整个页面。

新增功能：

1. Bridge health

   * 按钮：检查 App Server Bridge
   * 调用 `/app-server-bridge/health`
   * 显示 status / mode / sandbox / threads / error

2. App Thread 创建

   * 项目选择复用现有项目列表
   * 输入 title
   * 按钮：创建 App Thread
   * 调用 `POST /app-threads`

3. App Thread 列表

   * 调用 `GET /app-threads?limit=20`
   * 显示：

     * id
     * title
     * project_id
     * status
     * updated_at
   * 点击选择当前 App Thread

4. 发送 App Turn

   * 文本框输入 message
   * 按钮：发送 App Turn
   * 调用 `POST /app-threads/{id}/turns`
   * 显示 assistant_final

5. 查看 turns

   * 调用 `GET /app-threads/{id}/turns`
   * 显示最近 turns

6. 关闭 App Thread

   * 调用 `DELETE /app-threads/{id}`
   * 删除前 confirm

7. 安全：

   * 继续复用 localStorage 中的 `apiToken`
   * 请求主后端时使用 `X-API-Token`
   * 所有 innerHTML 动态字段继续使用 escapeHtml

不做：

* 不直接访问 8766 Bridge
* 不在主线 mobile 页面保存 Bridge Token
* 不做 SSE
* 不做审批 UI
* 不做 diff UI

v0.8.0 增加 App Server 会话模式主线集成：
