# 二、任务 1：新增一键启动脚本

新增文件：

```text
scripts/start_app_server_stack.ps1
```

目标：

一键启动当前 App Server 主线集成所需的两个服务：

```text
1. App Server Bridge sidecar：127.0.0.1:8766
2. 主后端 FastAPI：127.0.0.1:8000
```

## 脚本要求

使用 PowerShell 实现。

支持参数：

```powershell
-param(
  [string]$ApiToken = "dev-token",
  [string]$BridgeToken = "dev-token",
  [string]$BackendHost = "127.0.0.1",
  [int]$BackendPort = 8000,
  [string]$BridgeHost = "127.0.0.1",
  [int]$BridgePort = 8766,
  [string]$CodexCommand = "codex.cmd",
  [switch]$RunSmoke
)
```

## 行为

1. 在项目根目录执行。
2. 检查当前目录是否存在：

```text
backend/main.py
poc/app_server/app_server_bridge.py
```

如果不存在，输出错误并退出。

3. 设置环境变量：

```powershell
$env:API_TOKEN=$ApiToken
$env:APP_SERVER_BRIDGE_TOKEN=$BridgeToken
$env:APP_SERVER_BRIDGE_URL="http://$BridgeHost:$BridgePort"
```

4. 分别启动两个新 PowerShell 窗口：

```text
Bridge sidecar window
Backend API window
```

5. Bridge 启动命令：

```powershell
$env:APP_SERVER_BRIDGE_TOKEN="<BridgeToken>"
python .\poc\app_server\app_server_bridge.py --host <BridgeHost> --port <BridgePort> --codex-command <CodexCommand>
```

6. Backend 启动命令：

```powershell
$env:API_TOKEN="<ApiToken>"
$env:APP_SERVER_BRIDGE_URL="http://<BridgeHost>:<BridgePort>"
$env:APP_SERVER_BRIDGE_TOKEN="<BridgeToken>"
python -m uvicorn backend.main:app --host <BackendHost> --port <BackendPort>
```

7. 输出提示：

```text
主线 mobile:
http://127.0.0.1:8000/mobile

Bridge POC mobile:
http://127.0.0.1:8766/mobile

Smoke:
python .\scripts\smoke_app_server_flow.py --base-url http://127.0.0.1:8000 --project-path <当前项目路径>
```

8. 如果传入 `-RunSmoke`：

* 等待 3~5 秒。
* 执行：

```powershell
python .\scripts\smoke_app_server_flow.py --base-url http://127.0.0.1:8000 --project-path <当前项目路径>
```

9. 不要强依赖 Windows Terminal。
10. 使用 PowerShell 自带 `Start-Process powershell` 即可。

# 三、任务 2：增强 smoke 失败清理

修改文件：

```text
scripts/smoke_app_server_flow.py
```

当前问题：

如果 AppThread 创建成功，但后续 send turn / final / events 失败，脚本可能直接退出，残留 ACTIVE / ERROR AppThread。

## 修改要求

1. 在 `run_smoke_flow()` 中记录：

```text
created_app_thread_id
cleanup_status
```

2. 如果 AppThread 已创建，后续任一步失败，尽量调用：

```text
DELETE /app-threads/{id}
```

3. 清理失败也不要覆盖原始错误，但要在输出中显示：

```text
cleanup_status
cleanup_error
```

4. 成功流程仍然正常关闭 AppThread。

5. 失败输出 JSON 至少包含：

```json
{
  "pass": false,
  "failed_step": "...",
  "status_code": 503,
  "body": "...",
  "error": "...",
  "cleanup_status": "CLOSED | FAILED | SKIPPED",
  "cleanup_error": "..."
}
```

6. 如果 final 不包含 `app-thread-smoke-ok`，也要尽量关闭 AppThread，并返回 exit code 1。

7. 不要直接访问 Bridge 8766，只能访问主后端 8000。

# 四、任务 3：增加 AppThread 重建 Bridge Thread 能力

Bridge sidecar 重启后，旧的 `bridge_thread_id` 在 Bridge 内存中会丢失。此时主系统里 AppThread 还存在，但发送 App Turn 会失败。

本次增加一个最小恢复能力：

```text
POST /app-threads/{app_thread_id}/reopen
```

含义：

```text
为已有 AppThread 创建一个新的 Bridge thread，并把本地 AppThread 重新置为 ACTIVE。
```

## 修改文件

```text
backend/main.py
backend/services/app_thread_service.py
backend/schemas.py
backend/mobile.py
```

## Service 函数

新增：

```python
reopen_app_thread(session, app_thread_id, bridge_client=None)
```

行为：

1. 查询 AppThread，不存在返回 404。
2. 如果 AppThread.status 不是 `ERROR` 或 `CLOSED`，仍允许 reopen，但不要报错。
3. 调用 Bridge `create_thread(title=app_thread.title)`。
4. 要求返回新的 `bridge_thread_id`，复用 v0.8.1 的 `_require_bridge_thread_id()` 校验。
5. 更新：

```text
bridge_thread_id
app_thread_id
status=ACTIVE
last_error=None
updated_at=now
```

6. 不删除历史 AppTurn。
7. 不自动重发历史消息。
8. 返回 AppThreadRead。

## API

新增：

```text
POST /app-threads/{app_thread_id}/reopen
```

要求：

1. 需要 `X-API-Token`。
2. 成功返回 AppThreadRead。
3. Bridge 不可用时返回 502/503。
4. Bridge 返回缺 `bridge_thread_id` 时返回 502。

## Schema

如果需要，可新增：

```text
AppThreadReopenRead
```

但优先复用 `AppThreadRead`，减少复杂度。
