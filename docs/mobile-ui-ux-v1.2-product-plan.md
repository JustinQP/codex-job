# v1.2 Mobile App 产品化 UI/UX 第二版方案

## 1. 背景

当前 v1.1 已经完成第一轮 mobile UI/UX 重构：

- 页面从长页面改为四 Tab：任务 / App 会话 / Runner / 设置。
- 引入卡片、badge、toast、loading、折叠区。
- App 会话区具备气泡式展示。
- 任务创建和任务列表已初步卡片化。

但从真实手机 App 使用视角看，当前交互仍然偏“网页后台移动适配”，还不是“手机 App”：

- 按钮仍然太多、太大，操作密度不合理。
- 页面仍以“功能入口”组织，而不是以“用户当前任务”组织。
- 关键操作和次要操作没有充分分层。
- App 会话、任务详情、Runner 状态还停留在同屏堆叠。
- 缺少首页概览、详情页、底部操作栏、更多菜单、弹层等 App 常见交互。
- 用户容易被大量按钮打断，不知道下一步最应该做什么。

v1.2 的目标：

```text
把 mobile 从“移动端网页控制台”升级为“手机 App 风格的 Codex 工作台”。
```

---

## 2. 产品设计原则

### 2.1 一个页面只解决一个核心任务

不要在一个页面里同时放创建、列表、详情、调试、设置、危险操作。

页面应该按用户意图拆分：

```text
首页：看状态 + 快速进入
任务：创建任务 / 查看任务
会话：和 App Server 交互
运行：看 Runner 和系统状态
设置：低频配置
```

### 2.2 主操作露出，次操作收进菜单

每个页面最多保留 1 个主按钮。

次操作进入：

```text
- 更多菜单
- 底部弹层 Action Sheet
- 折叠详情
- 长按/更多按钮
```

### 2.3 手机上减少“大按钮堆叠”

按钮不应铺满一屏。

推荐：

```text
- 主按钮：页面底部固定 1 个
- 次按钮：小型文本按钮 / icon button
- 危险操作：更多菜单 + confirm
- 批量/低频操作：设置或管理页
```

### 2.4 默认展示结果，不默认展示调试信息

用户关心：

```text
- 当前状态
- 是否成功
- 结果是什么
- 还能做什么
```

不默认展示：

```text
- raw JSON
- bridge_thread_id
- event_counts
- 调试日志
```

调试信息统一放到“详情 / 调试”折叠页。

### 2.5 先做减法，再做美化

v1.2 的核心不是换颜色，而是：

```text
减少按钮
减少表单
减少同屏信息量
减少无意义滚动
减少误操作
```

---

## 3. 新的信息架构

建议从四 Tab 调整为五个一级入口，采用 App 常见结构。

```text
底部导航：
1. 首页
2. 任务
3. 会话
4. 运行
5. 我的
```

如果担心 5 个 Tab 太多，可以保留 4 个：

```text
首页
任务
会话
我的
```

其中 Runner 状态并入首页和“我的/运行诊断”。

推荐最终采用：

```text
首页 / 任务 / 会话 / 我的
```

理由：

- Runner 对普通使用不是一级目标。
- 用户打开 App 首先想知道“现在能不能用、有没有任务在跑”。
- 设置和 Runner 都是低频，不应占一个底部 Tab。

## 4. 页面设计

# 4.1 首页 Tab

## 目标

首页不是操作全集，而是“状态总览 + 快速入口”。

## 页面结构

```text
顶部：
- Codex 工作台
- 当前连接状态

状态卡片：
- Backend：正常/异常
- Runner：在线 x 个 / 离线 x 个
- Bridge：正常/异常
- 当前运行任务：x 个
- 当前运行 AppTurn：x 个

快捷入口：
- 新建任务
- 打开最近任务
- 打开最近会话

最近活动：
- 最近 3 个任务
- 最近 3 个会话 / AppTurn
```

## 交互要求

- 首页只保留“新建任务”一个主按钮。
- 最近任务和最近会话点击进入详情。
- Runner/Bridge 异常时，首页展示一条红色提示卡。
- 不在首页放所有操作按钮。

## 技术说明

当前后端可能没有专门 dashboard API。

v1.2 第一版不要新增后端 API，可以用现有接口组合：

```text
GET /health
GET /runners
GET /tasks?limit=3
GET /app-server-bridge/health
GET /app-threads?limit=3
```

# 4.2 任务 Tab

## 目标

任务 Tab 专注主线 Runner/codex exec。

不要同时把创建表单、任务列表、任务详情、日志预览全堆在同一屏。

## 页面结构

```text
顶部：任务
- 筛选：全部 / 运行中 / 成功 / 失败
- 小型刷新 icon

任务列表：
- 卡片列表
- 状态 badge
- 标题 / task_type
- 更新时间
- 简短结果预览

底部固定主按钮：
+ 新建任务
```

## 新建任务交互

点击“+ 新建任务”后，不在当前页面展开大表单，而是打开底部弹层或全屏表单。

```text
新建任务 Sheet / 页面：
1. 选择项目
2. 输入 Prompt
3. 选择任务类型
4. 提交

高级设置：折叠
- Runner
- 模型
- 推理难度
- sandbox
- timeout
```

## 任务详情交互

点击任务卡片进入任务详情页/详情面板：

```text
任务详情：
- 状态
- Prompt
- 运行参数
- 日志预览
- 结果预览
- 操作栏：取消 / 重跑 / 查看 log / 查看 result / 查看 diff
```

操作栏设计：

```text
主操作：查看结果
次操作：log / diff / 重跑 / 取消
危险操作：取消必须 confirm
```

## 按钮密度要求

任务列表卡片不要放 5 个并排大按钮。

建议：

```text
卡片右上角：更多 ...
卡片底部最多两个文本入口：详情 / 结果
其他操作进入更多菜单
```

---

# 4.3 会话 Tab

## 目标

会话 Tab 应该像聊天 App，而不是后台调试面板。

## 页面结构

```text
顶部：
- 当前会话标题
- 状态 badge
- 更多 ...

中间：
- 消息流

底部：
- 输入框
- 发送按钮
- 可切换同步/异步
```

## AppThread 列表

不要默认显示一大串 AppThread。

改为顶部标题点击后进入：

```text
会话列表 Sheet：
- 搜索/筛选
- 最近会话
- 新建会话
- archived 开关
```

或者在顶部加一个小按钮：

```text
切换会话
```

## 当前会话更多菜单

顶部更多菜单包含：

```text
- 重命名
- 重开会话
- 关闭会话
- 查看事件摘要
- 调试信息
- 恢复卡住 turn
```

不要把这些按钮常驻在主屏。

## 发送方式

同步/异步不要做两个同等大按钮。

建议：

```text
默认发送：异步
发送按钮旁边有一个小切换：同步/异步
```

或者：

```text
长按发送按钮：选择同步发送
默认点击：异步发送
```

为了实现简单，v1.2 可以做：

```text
[异步] toggle + [发送]
```

## AppTurn 气泡

气泡展示原则：

```text
用户消息：右侧
助手回复：左侧
运行中：左侧 loading
失败：左侧红色错误卡
取消：左侧灰色卡
```

每个气泡不要展示太多 meta。

默认只展示：

```text
内容
状态
耗时
```

更多信息点击气泡展开：

```text
- created_at
- bridge_turn_id
- event summary
- raw summary
```

---

# 4.4 我的 / 设置 Tab

## 目标

低频设置和系统说明集中放这里。

## 页面结构

```text
Token 设置
系统状态
Runner 管理
Bridge 说明
Smoke 命令
当前限制
关于版本
```

Runner 不再独立占底部 Tab。

Runner 状态以列表卡片展示：

```text
Runner ID
状态
hostname
pid
models
last heartbeat
```

低频操作：

```text
刷新 Runner
检查 Bridge
recover stale
cleanup archived
```

这些操作不应该出现在主会话页的显眼位置。

## 5. 关键交互模式

# 5.1 底部弹层 Action Sheet

建议新增通用弹层：

```javascript
openSheet(title, contentHtml)
closeSheet()
```

用于：

```text
新建任务
任务更多操作
切换会话
会话更多操作
事件摘要
调试输出
```

如果实现成本太高，v1.2 第一版可以用 `<dialog>` 或自定义 fixed div。

# 5.2 更多菜单

所有低频操作收进“...”菜单：

任务卡片：

```text
查看详情
查看日志
查看结果
查看 diff
重跑
取消
```

会话顶部：

```text
切换会话
重命名
重开
关闭
事件摘要
调试输出
恢复卡住 turn
```

# 5.3 主按钮策略

每个主页面最多一个主要按钮：

```text
首页：新建任务
任务：新建任务
会话：发送
我的：保存 Token
```

其他按钮降级为：

```text
icon button
文本按钮
更多菜单
详情弹层
```

# 5.4 空状态设计

空状态要告诉用户下一步怎么做。

例如：

任务空状态：

```text
还没有任务
点击下方「新建任务」开始第一次 Codex 执行
```

会话空状态：

```text
还没有 App 会话
点击「新建会话」开始一次 App Server 对话
```

Runner 空状态：

```text
没有在线 Runner
请先启动 runner/runner.py 或 scripts/start.bat runner
```

# v1.2.0：App 化信息架构与首页

## 目标

把底部导航从“功能分类”调整成“用户心智分类”。

从：

```text
任务 / App 会话 / Runner / 设置
```

改为：

```text
首页 / 任务 / 会话 / 我的
```

## 任务范围

1. 新增首页 Tab。
2. Runner Tab 移入“我的”页。
3. 设置 Tab 改名“我的”。
4. 首页展示系统状态和最近活动。
5. 首页只保留一个主按钮：新建任务。
6. 保留原 API，不新增后端。

## 首页展示

```text
- Backend 状态
- Runner 在线数量
- Bridge 状态
- 最近任务 3 个
- 最近会话 3 个
```

## 验收标准

```text
1. 底部导航为：首页 / 任务 / 会话 / 我的。
2. Runner 不再作为独立一级 Tab。
3. 首页能看到系统状态和最近活动。
4. 首页没有大按钮堆叠。
5. 原任务、会话、Runner、设置功能入口不丢失。
6. pytest -q 通过。
```

# v1.2.1：任务流重构

## 目标

让任务页像一个任务 App，而不是表单页。

## 任务范围

1. 任务页默认只展示任务列表。
2. 新建任务改为底部弹层 / 全屏表单。
3. 任务卡片只保留：详情 / 结果 / 更多。
4. log/result/diff/取消/重跑进入更多菜单或详情页。
5. 任务详情从主页面剥离成详情面板。

## 验收标准

```text
1. 任务列表不再每张卡片放 5 个大按钮。
2. 新建任务不会撑满任务列表页面。
3. 高级参数默认折叠。
4. 任务详情层级清晰。
5. 取消任务必须 confirm。
6. pytest -q 通过。
```

---

# v1.2.2：会话流重构

## 目标

让 App 会话页更像聊天 App。

## 任务范围

1. AppThread 列表默认隐藏，通过“切换会话”进入。
2. 当前会话顶部只显示标题、状态、更多菜单。
3. 低频操作进入更多菜单。
4. 发送区改为一个发送按钮 + 同步/异步 toggle。
5. 气泡 meta 默认弱化，点击气泡展开详情。
6. event summary / debug output 进入详情弹层。

## 验收标准

```text
1. 会话页主屏只看到当前会话、消息流和输入框。
2. 不再常驻一堆 reopen/close/recover/events/debug 按钮。
3. 默认发送方式清晰。
4. 异步运行状态明显。
5. 错误/取消/成功状态清晰。
6. pytest -q 通过。
```

## 7. 本次不做

v1.2 仍然不做：

```text
1. 不引入 Vue/React。
2. 不引入前端构建。
3. 不新增后端 API，除非非常必要。
4. 不做 SSE。
5. 不做审批 UI。
6. 不做 diff UI。
7. 不做多用户权限。
8. 不做公网部署。
9. 不改 Runner/codex exec 主链路。
10. 不改 App Server Bridge 架构。
```

## 9. 最终目标

v1.2 完成后，mobile 应该从：

```text
带 App 样式的移动网页
```

升级为：

```text
真正按手机 App 使用心智设计的 Codex 工作台
```

判断标准：

```text
用户打开后先看到状态和下一步，而不是一堆按钮。
用户执行任务时只看到任务相关内容。
用户对话时只看到会话相关内容。
低频、危险、调试操作不会干扰主流程。
```
