# 3. P0：会破坏核心功能的问题

## P0-1 Agent 重试会重复执行业务命令

### 代码证据

`agent/command_loop.py` 中：

```text
run_once()
  -> _with_retries(_run_once)
```

而 `_run_once()` 包含完整流程：

```text
heartbeat
claim
ack
renew
handler.handle(command)
complete
clear local state
```

涉及：

```text
agent/command_loop.py:92-141
agent/command_loop.py:144-155
```

本地状态只保存：

```text
command_id
claim_request_id
lease_token
```

见：

```text
agent/local_state.py:14-18
```

没有保存：

```text
执行阶段
handler 是否已执行
本地执行结果
complete 是否待重试
```

### 失败场景一：服务端已完成，响应丢失

```text
Agent 执行 Codex 成功
-> POST /complete 到达服务端
-> 服务端设为 SUCCESS 并清除 lease
-> HTTP 响应丢失
-> Agent 重跑整个 _run_once
-> claim 返回旧命令
-> Agent 使用旧 lease 再 ack
-> invalid_lease_token
-> Agent 卡住，不能领取下一条命令
```

### 失败场景二：complete 未到服务端

```text
Agent 执行 Codex 成功
-> complete 网络失败
-> Agent 重跑整个 _run_once
-> handler 再执行一次
```

### 影响

- `RUN_EXECUTE` 可能重复修改代码
- `TURN_START` 可能向同一 thread 重复发送消息
- `SESSION_OPEN` 可能创建多个 App Server 进程
- Agent 可能永久卡在旧 command

### 修复方案

Agent 本地状态应升级为：

```text
CLAIMED
EXECUTING
RESULT_READY
COMPLETION_PENDING
COMPLETED
```

执行结束后持久化：

```json
{
  "command_id": "xxx",
  "phase": "COMPLETION_PENDING",
  "result": {
    "success": true,
    "error_message": null,
    "result_payload": {}
  }
}
```

重试时：

- `EXECUTING`：执行恢复或明确失败，不盲目重跑
- `COMPLETION_PENDING`：只重试 complete
- 服务端已终态：清理本地状态
- claim 返回终态命令：不再 ack

### 必补测试

1. complete 响应丢失，handler 只执行一次
2. complete 请求未到服务端，只重发 complete
3. Agent 重启后恢复 completion pending
4. TURN_START 不重复发送
5. SESSION_OPEN 不重复创建进程

## P0-2 取消会清除 lease，Agent 因此看不到取消

### 代码证据

控制端取消通过：

```text
request_cancel_command()
```

直接把命令转换为终态 `CANCELLED`。

终态转换会清除：

```text
lease_token
lease_expires_at
```

Agent 的取消检测却是：

```text
renew_command()
reconcile()
```

见：

```text
agent/run_executor.py:95-115
agent/session_handlers.py:120-139
```

renew 失败后被捕获并返回 `False`，不会继续 reconcile。

### 实际结果

```text
用户点击取消
-> 控制端命令变为 CANCELLED
-> lease 被清空
-> Agent renew 失败
-> should_cancel 返回 False
-> Codex 继续运行
```

Run 取消还会立即释放 Workspace 锁：

```text
backend/services/run_service.py:225-244
```

因此可能出现：

```text
旧 Codex 仍在写
-> 服务端锁已释放
-> 新写入执行获得锁
-> 同一目录并发修改
```

### 正确模型

增加：

```text
cancel_requested
cancel_requested_at
```

状态流转：

```text
RUNNING
  -> cancel_requested=true
  -> Agent 检测
  -> Agent 终止进程
  -> Agent complete(CANCELLED)
  -> 服务端设置终态并释放锁
```

取消查询不应依赖 renew 成功。

## P0-3 Run 日志与产物未上传

### 代码证据

`RunExecutor` 只在 Agent 本机创建：

```text
data/agent-runs/<run_id>/run.log
data/agent-runs/<run_id>/result.md
data/agent-runs/<run_id>/diff.patch
```

见：

```text
agent/run_executor.py:64-87
```

仓库虽然存在：

```text
agent/log_uploader.py
agent/artifact_uploader.py
```

但 `RunExecutor` 没有接入它们，也没有使用 `on_tick` 上传日志。

控制端在 command complete 后直接将 Run 标为 SUCCESS。

### 影响

- UI 显示 SUCCESS
- `/runs/{id}/log` 可能 404
- `/runs/{id}/result` 可能 404
- `/runs/{id}/diff` 可能 404
- Agent 本地文件与服务端状态分裂

### 修复方案

```text
execute_codex(on_tick=upload_log_chunk)
-> 上传最后日志
-> build artifact manifest
-> 上传 result/diff/git status/report
-> 确认必要产物上传完成
-> complete SUCCESS
```

上传失败时只重试上传，不重复执行 Codex。

### 附带问题

`RunLogUploadTracker.next_chunk()` 会把 offset 后的全部内容作为一个 chunk，但服务端限制 64 KiB。必须按服务端最大值分片。

## P0-4 AppTurn final 事件序号冲突

### 代码证据

`run_turn()`：

```text
status sequence=1
外层 sequence 变为 2
_wait_for_turn_completed(first_sequence=2)
```

内部事件循环会递增自己的 sequence，但只返回 `end_index`。

回到外层后，final 仍使用旧的 `sequence=2`。

涉及：

```text
agent/app_server/session_manager.py:183-192
agent/app_server/session_manager.py:219-264
agent/app_server/session_manager.py:288-323
```

### 影响

只要过程事件已经占用 sequence=2，final 再上传 sequence=2 会触发：

```text
command_event_sequence_conflict
```

于是：

- Codex turn 实际已完成
- handler 返回失败
- AppTurn 被标记 FAILED
- AppThread 进入 RECOVER_REQUIRED

### 修复方案

让事件循环返回：

```python
(end_index, next_sequence)
```

或增加统一、持久化的 `EventSequenceAllocator`。

## P0-5 关闭会话不会关闭 Agent App Server

### 代码证据

控制端 `close_app_thread()` 只：

```text
status = CLOSED
release control-plane workspace lock
```

没有创建 `SESSION_CLOSE` 命令。

Agent Session 只有显式 `close_session()` 才会：

- 关闭 app-server process
- 释放本地 Workspace lock

### 影响

- App Server 子进程泄漏
- Agent 本地 Session 泄漏
- 本地写锁不释放
- reopen 或新写入 Session 可能持续 workspace busy
- 控制端与 Agent 锁状态不一致

### 修复方案

新增：

```text
SESSION_CLOSE
```

流程：

```text
用户关闭
-> 创建 SESSION_CLOSE
-> Agent close_session
-> 关闭进程
-> 释放本地锁
-> complete SUCCESS
-> 控制端标记 CLOSED 并释放服务端锁
```

`AgentAppSessionManager.close_all()` 也必须释放所有 lock callback，并在 Agent 退出 `finally` 中调用。

# 4. P1：高优先级问题

## P1-1 SESSION_OPEN 失败时 Thread 可能卡在 OPENING

Agent complete 路由只有在：

```text
payload.result_payload is not None
```

时才调用 `complete_agent_session_open()`。

SessionOpen 失败通常返回：

```text
success=False
result_payload=None
```

命令已经 FAILED，但 AppThread 不会被更新为 ERROR。

涉及：

```text
backend/routers/agent.py:174-186
backend/services/app_thread_service.py:130-156
```

修复：无论成功失败都调用 completion service；成功但缺少 session/thread ID 时也设为 ERROR。

---

## P1-2 Run 不会在 ACK 时变为 RUNNING

ACK 路由只处理：

```text
TURN_START -> mark_agent_turn_running()
```

没有：

```text
RUN_EXECUTE -> mark_run_running()
```

后果：

- 执行中仍显示 PENDING
- started_at 不准确
- 取消按 PENDING 路径提前释放锁
- 运行监控不可信

---

## P1-3 临时事件上传失败会中止 Turn

App Session Manager 每条事件都立即：

```text
cache_event()
flush()
```

`flush()` 网络异常会直接向上抛出。

结果：

```text
Codex 正在运行
-> Control Plane 短暂不可达
-> flush 失败
-> handler 返回 FAILED
-> App Server 可能仍在后台继续执行
```

本地 pending cache 应让上传失败降级，而不是终止 Codex。

建议：

- cache 成功后继续
- flush 失败只保留 pending
- 独立后台 uploader 分批重试
- 大于 100 条事件分批
- 超过 16 KiB 的 raw event 做切分或摘要

---

## P1-4 过期 Command 没有服务端主动回收

`_handle_expired_lease()` 只在旧命令再次调用 lease 校验时触发。

`claim_command()` 只找 PENDING，不扫描：

```text
CLAIMED/RUNNING + lease_expires_at < now
```

Agent 永久宕机后：

- Command 长期卡在 CLAIMED/RUNNING
- Run/Turn 无终态
- Workspace lock 可能长期占用

应增加 sweeper，并同步更新领域对象和锁。

---

## P1-5 Workspace 锁固定 6 小时且不续租

```text
WORKSPACE_LOCK_LEASE_SECONDS = 6 * 60 * 60
```

连续 Session 建立后没有续租。

长于 6 小时后，服务端可能允许新的写入执行。应由 Session 活动、Agent heartbeat 或 Turn renew 同步续租。

---

## P1-6 长任务期间 Device 会被标记 OFFLINE

Device lease 为 60 秒。

Agent heartbeat 只在主循环每轮开始执行，handler 同步阻塞期间不会再次 heartbeat。Run/Turn 只 renew Command，不 heartbeat Device。

执行超过 60 秒后，读取 `/devices` 就可能把正在工作的设备标记 OFFLINE。

建议使用独立 heartbeat 线程。

---

## P1-7 Session 不能跨 Agent 重启恢复

`AgentAppSessionManager._sessions` 只在内存。

Agent startup reconciliation 只检查当前 command，不恢复 Session。

Agent 重启后：

- Control Plane 仍可能显示 ACTIVE
- 下一条 Turn 才发现 `agent session not found`
- 随后才进入 RECOVER_REQUIRED

应在 Agent 启动时主动上报 Session 状态，不要等用户发消息后才暴露问题。

---

## P1-8 migration/reset 策略会静默留下不兼容旧库

当前：

```python
MIGRATIONS = ()
```

并且 `run_migrations()` 忽略：

```text
db_path
backup
```

`create_all()` 不会更新已有表字段。

旧 `app.db` 可能：

```text
后端启动
-> 首次查询新字段
-> no such column
```

即使开发期允许删除旧数据，也应：

- 检测 schema generation
- 发现旧结构时 fail-fast
- 输出明确 reset 命令
- 不保留“会迁移、会备份”的假象

---

## P1-9 Project 仍要求 Control Plane 本地路径存在

`create_project()` 仍执行：

```text
Path(payload.path).resolve()
exists()
is_dir()
```

多设备场景：

```text
Control Plane 在 A
目录在 B
-> A 上路径不存在
-> Project 无法创建
```

其他不一致：

- `ProjectCreate` 不能传 workspace_id
- `bind_project_workspace()` 没有公开 Router
- Run/AppThread 接受任意 project_id + workspace_id
- 未校验两者绑定关系

建议把 Project 收敛为 Workspace 上的逻辑配置，或合并概念。

---

## P1-10 Run/Session 创建事务不完整

`create_run()` 在创建 AgentCommand 前已经多次 commit。

后续 command 创建或锁获取失败，会留下 PENDING Run。

Session 创建也由多个 service 各自 commit，失败时可能留下没有 command 的 OPENING AppThread。

建议：

```text
单业务动作 = 单事务
flush 获取 ID
全部成功后统一 commit
失败统一 rollback
```

# 5. P2：建议优化

## P2-1 Turn 列表仍取最早 N 条

`list_app_turns()` 使用：

```text
ORDER BY id ASC
LIMIT N
```

长会话超过 limit 后，最新 Turn 不返回。应 DESC 取最新 N，再按 ASC 返回。

## P2-2 AppThread 列表 N+1

每个 Thread 单独查 latest success 和全部 Turn 计数。应使用聚合或批量查询。

## P2-3 App Server executable 硬编码为 `codex.cmd`

Run 链路支持 `CODEX_BIN` 查找，Session 链路默认固定 `codex.cmd`。应统一配置。

## P2-4 Agent 本地状态不是原子写

identity/state 直接覆盖 JSON。崩溃可能产生半写文件。应 tmp + fsync + atomic replace，并保留备份。

## P2-5 Artifact 对非 UTF-8 内容可能校验失败

Agent 对原始 bytes 计算 size/hash，再用 `errors=replace` 解码；服务端重新编码后验证。非 UTF-8 内容会不一致。建议 Base64，或先规范化 UTF-8 再计算 hash。

## P2-6 CI 路径落后

CI 仍执行：

```text
python -m compileall backend runner scripts poc/app_server
```

但正式代码已是 `agent/`，旧目录已删除。建议：

```text
python -m compileall backend agent scripts
pytest -q
```

并确认最新 HEAD 有实际 workflow run。

## P2-7 共享 Agent Token 可伪装 device_id

个人可信网络场景下可以暂时接受，但应：

- 不把 AGENT_TOKEN 与 API_TOKEN 配成相同值
- 只部署在可信私网
- 后续可做每设备 Token，不必引入租户体系

# 6. 安全审查

## 已做得较好

### Workspace 本地白名单

Agent 仅通过本地 Registry 解析 workspace key，并验证路径位于 allowed roots。

### Command payload 禁止绝对路径

### Artifact 路径保护

读取 Artifact 时验证路径必须位于 Control Plane `JOBS_DIR`。

### API Token 与 Agent Token 分离

符合个人工具的最小安全模型。

## 仍需注意

- `API_TOKEN` 未配置时业务 API 完全开放
- README 使用 `0.0.0.0` 启动，必须配合可信网络和 Token
- Agent upload log/artifact 没有验证 active lease
- 取消后晚到事件和产物需要明确接受/拒绝规则
- 共享 Agent Token 允许设备身份伪装

# 9. 发布判断

## 当前可以做

- 继续内部开发
- 使用 Fake Agent 验证调度
- read-only 短会话试用
- 人工监督下运行低风险任务

## 当前不建议

- 长时间无人值守 workspace-write
- 依赖“取消”保护真实项目
- 假设 SUCCESS 一定有完整日志和产物
- 让连续 Session 跨 Agent 重启长期使用
- 无 CI 结果继续批量合并核心状态机修改

## 可长期自用的最低门槛

- P0-1 至 P0-5 全部关闭
- Run 日志与产物完整闭环
- 取消实际终止进程
- Session close/reopen 无进程和锁泄漏
- 断网不重复执行
- 最新 HEAD CI 全绿
- 双真实设备 smoke 通过
