## 1. 定位

本文件用于治理项目文档，防止 docs 目录发散、文档重复、命名混乱和过程文件堆积。

适用于以下任务：

* 新增文档
* 修改文档
* 合并文档
* 归档文档
* 调整 docs 目录
* 生成技术方案
* 生成开发计划
* 生成复盘材料
* 生成工程规范

目标是让文档成为项目资产，而不是过程垃圾桶。

## 2. 基本原则

文档必须服务于项目开发、维护、交付或复盘。

新增文档前必须先判断：

* 是否真的需要新建文档
* 是否已有同类文档
* 是否可以更新已有文档
* 是否可以合并到已有文档
* 文档应该放在哪个目录
* 文件名是否清晰稳定
* 是否会造成重复或冲突

禁止为了记录过程而无限新增文档。

## 3. 文档类型

新增或修改文档前，应先判断文档类型。

常见类型包括：

### 项目说明

用于说明项目背景、目标、快速开始、部署方式等。

示例：

```text
README.md
快速开始.md
部署说明.md
项目概览.md
```

### 架构设计

用于说明系统结构、模块设计、接口设计、数据模型等。

示例：

```text
系统架构设计.md
模块设计.md
接口设计.md
数据库设计.md
```

### 开发计划

用于说明版本规划、开发计划、任务拆分、迭代安排。

示例：

```text
v0.3-开发计划.md
v0.4-任务拆分.md
版本路线图.md
```

### 规则规范

用于说明工程规范、协作规则、AI 规则、编码约定。

示例：

```text
编码规范.md
文档治理规则.md
AI协作规则.md
测试验收规则.md
```

### 问题复盘

用于记录故障分析、踩坑记录、优化建议、项目复盘。

示例：

```text
2026-06-项目复盘.md
接口超时问题复盘.md
部署故障分析.md
```

### 交付材料

用于沉淀汇报、验收、方案、客户交付材料。

示例：

```text
技术方案.md
验收材料.md
汇报提纲.md
交付清单.md
```

## 4. docs 目录结构

如果项目存在 docs 目录，应保持目录结构稳定。

推荐结构：

```text
docs/
  00-overview/        # 项目概览、背景、目标、快速开始
  10-design/          # 架构设计、模块设计、接口设计、数据设计
  20-plan/            # 版本计划、开发计划、任务拆分、迭代记录
  30-rules/           # 工程规范、协作规则、AI 规则、测试规则
  40-review/          # 项目复盘、问题记录、优化建议、故障分析
  90-archive/         # 归档内容、历史方案、废弃文档
```

新增文档时，应优先放入上述目录。

如果当前项目已有自己的 docs 结构，应优先沿用现有结构，不要强行替换。

禁止直接在 docs 根目录持续堆放大量零散文档。

## 5. 文档新增规则

新增文档必须满足：

* 有明确用途
* 有明确归属目录
* 命名清晰稳定
* 与现有文档不重复
* 与现有项目结构一致
* 内容不是纯过程草稿
* 后续仍有维护价值

新增文档前，应先搜索或检查是否已有相关文档。

如果已有同类文档，应优先更新已有文档。

## 6. 文档命名规则

文档命名应清晰、稳定、可长期维护。

推荐命名方式：

```text
主题-用途.md
版本号-主题.md
日期-主题.md
```

示例：

```text
v0.3-开发计划.md
目录生成-技术方案.md
2026-06-项目复盘.md
AI协作规则.md
测试验收规则.md
```

禁止使用含义模糊的文件名，例如：

```text
新建文档.md
方案.md
优化.md
记录.md
临时.md
最终版.md
最终最终版.md
最新.md
新版.md
副本.md
```

如果文档是临时过程材料，应放入明确的计划目录或归档目录，不要散落在项目根目录。

## 7. 文档更新规则

修改文档时，应优先保持文档主线清晰。

如果只是补充已有内容，应直接更新原文档。

如果是替代旧方案，应明确标记：

* 当前推荐方案
* 历史方案
* 废弃原因
* 迁移说明

禁止在不说明关系的情况下，同时保留多份互相冲突的方案文档。

## 8. 文档合并规则

如果发现多个文档职责重复，应优先考虑合并。

适合合并的情况：

* 标题不同但内容高度相似
* 多个文档都在描述同一方案
* 同一版本有多个计划文件
* 多个复盘文件记录同一个问题
* 新旧方案并存但没有说明关系

合并时应保留：

* 当前有效内容
* 必要历史信息
* 关键决策原因
* 废弃内容说明

合并后，旧文档应移动到归档目录，或在索引中标记废弃关系。

## 9. 文档归档规则

以下文档应归档到 `docs/90-archive/`：

* 旧版本方案
* 已废弃设计
* 临时过程文档
* 被新文档替代的计划
* 已完成但仍有参考价值的历史材料

归档文档应说明：

* 归档原因
* 被哪个文档替代
* 是否仍有参考价值

禁止直接删除重要历史文档，除非用户明确要求。

## 10. 文档根目录规则

项目根目录只应保留少量高频入口文档。

通常允许：

```text
README.md
AGENTS.md
CHANGELOG.md
LICENSE
```

不建议在项目根目录堆放：

```text
开发计划.md
临时方案.md
接口设计.md
优化记录.md
测试记录.md
复盘.md
```

这些文档应进入 docs 对应目录。

## 11. AI 生成文档规则

AI 生成文档时必须注意：

* 不要为了过程记录创建文档
* 不要重复创建同主题文档
* 不要使用“最终版”等不可维护命名
* 不要生成与项目结构无关的目录
* 不要在没有用户要求时大范围重写文档体系
* 不要把草稿当成正式文档
* 不要同时保留多个互相冲突的方案

如果只是给用户看一份方案，不一定要写入项目文件。

## 12. 文档修改总结

如果任务涉及文档修改，完成后必须说明：

```text
文档变更：
- 新增：
- 修改：
- 归档：
- 删除：

文档位置：
-

是否存在旧文档需要合并或归档：
-
```

如果没有新增文档，应明确说明“未新增文档”。

## 13. 禁止事项

禁止以下行为：

* 在 docs 根目录持续堆放零散文档
* 创建多个职责重复的方案文档
* 使用“最终版”“临时版”等不可维护命名
* 不检查已有文档就新增文档
* 为了记录过程创建大量文档
* 不说明关系就保留多个冲突方案
* 随意删除历史文档
* 强行替换项目已有 docs 结构
* 把草稿文件放在项目根目录
