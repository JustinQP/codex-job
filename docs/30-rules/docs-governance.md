# 文档治理规则

## 1. 目标

文档应帮助当前开发、运行、维护和复盘，而不是成为过程文件堆积区。

本项目是个人工具，文档结构保持简单。只有确实存在长期维护价值时才新增文件或目录。

## 2. 文档类型

### 当前说明

用于当前有效的产品、接口、架构、运行和验收信息，例如：

```text
README.md
docs/api-overview.md
docs/app-server-session.md
docs/state-machines.md
docs/smoke-checklist.md
```

### 活跃计划

用于跨多个提交或阶段的正式开发工作，统一放在：

```text
docs/20-plan/
```

同一产品方向最多保留：

- 一份路线图：目标、范围、架构方向、阶段和完成标准。
- 一份执行清单：可独立执行的任务、依赖和验收标准。

一次性小改动不新建 plan，应直接在任务描述或对话中说明。

### 工程规则

统一放在：

```text
docs/30-rules/
```

规则文件职责必须单一，不允许相同内容以不同文件名重复存在。

### 历史资料

已完成、被替代或不再作为当前依据的计划进入：

```text
docs/90-archive/
```

个人项目允许使用“归档索引 + 不可变 Git commit 链接”保存历史原文，避免把大量完成计划复制到当前工作树。重要历史必须能通过归档索引或 Git 历史找回。

## 3. 新建规则

新增文档前必须确认：

1. 是否已有同主题文档可更新。
2. 是否需要跨多个提交持续引用。
3. 是否记录长期有效的架构、接口、运行或验收信息。
4. 是否只是临时分析、提示词或一次性步骤。
5. 文件应该属于当前说明、活跃计划、规则还是历史归档。

以下内容通常不提交：

- 临时头脑风暴。
- 单次小改动计划。
- Codex 中间分析过程。
- 本地运行日志。
- 与已有计划重复的新版本计划。
- 无长期参考价值的完成 checklist。

## 4. 命名规则

文件名使用稳定主题，不使用：

```text
最终版
最终最终版
最新版
新版
副本
临时
plan-2
```

推荐：

```text
<主题>-roadmap.md
<主题>-task-list.md
<主题>-design.md
<主题>-runbook.md
<版本>-retrospective.md
```

历史文件名存在错误时，在归档索引中说明，不为了修饰历史而修改原 commit。

## 5. 目录规则

当前推荐结构：

```text
docs/
  README.md
  api-overview.md
  app-server-session.md
  state-machines.md
  smoke-checklist.md
  20-plan/
  30-rules/
  90-archive/
```

根目录只保留少量高频、长期有效文档。只有同类文档明显增多时，再增加：

```text
00-overview/
10-design/
40-review/
```

禁止为了形式一次性建立大量空目录。

## 6. 更新规则

- 同一主题优先更新原文件。
- 架构或接口发生变化时同步更新对应当前说明。
- 文档移动时同步更新 README、测试、脚本和交叉链接。
- 当前文档不得长期描述已经失效的版本行为。
- 历史 plan 不作为当前实现的唯一依据。

## 7. 计划生命周期

### 创建

正式计划必须包含：

- 背景和目标。
- 范围与不做事项。
- 依赖和执行顺序。
- 验收标准。
- 测试方式。
- 风险与回滚。

### 执行

- 路线图维护阶段状态，不记录大量运行日志。
- Codex 任务清单一次只执行一个任务 ID。
- 完成任务可以更新 checkbox 和简短执行结果。
- 不为每一个小任务再创建独立 plan 文件。

### 完成

版本或路线完成后：

1. 将仍有效的内容提炼到 README、设计、API、runbook 或规则。
2. 必要时增加一份复盘，而不是保留多份过程计划。
3. 从 `20-plan/` 移除已经完成的计划。
4. 在 `90-archive/README.md` 中记录固定 commit 链接和归档说明。

## 8. 测试与文档

- 测试应验证当前接口、代码和有效文档。
- 不应通过读取历史 plan 文案来证明当前功能存在。
- 文档结构测试可以校验：
  - 必要当前文档存在。
  - 活跃计划数量受控。
  - 历史 plan 没有重新堆回根目录。
  - 归档索引可发现历史资料。

## 9. 文档变更输出

涉及文档整理时，完成总结必须说明：

```text
文档变更：
- 新增：
- 修改：
- 归档：
- 删除：

当前有效入口：
-

历史资料位置：
-

测试与引用更新：
-
```

## 10. 禁止事项

- 不检查已有文件就新增同主题文档。
- 在 `docs` 根目录持续增加版本计划。
- 同时维护多份互相冲突的路线图。
- 把临时提示词和中间分析作为正式文档提交。
- 无索引地删除重要历史资料。
- 文档移动后保留失效链接和失效测试。
- 用历史 plan 代替当前架构、接口或运行说明。
