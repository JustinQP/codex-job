## 1. 定位

本文件用于约束项目测试与验收流程。

适用于以下任务：

* 功能开发
* Bug 修复
* 接口修改
* 数据库修改
* 配置修改
* 前端页面修改
* AI 功能修改
* 部署相关修改
* 文档或规则修改

目标是让每次修改都有明确验证证据，而不是只说“已完成”。

## 2. 基本原则

项目修改完成后，不能只说明“已完成”，必须提供可验证的结果。

所有开发任务都必须经过以下闭环：

1. 明确验收标准
2. 执行必要测试
3. 记录验证命令或验证方式
4. 说明验证结果
5. 暴露风险与未完成项

禁止伪造测试结果。

禁止把未执行的测试写成已通过。

## 3. 验收标准

开始开发前，应尽量明确验收标准。

验收标准应具体、可检查。

禁止只写：

```text
功能正常
体验良好
代码没问题
测试通过
```

推荐写法：

```text
用户提交合法参数后，可以成功创建任务
参数缺失时返回明确错误信息
任务失败时状态更新为 failed，并记录失败原因
重复提交不会产生重复数据
现有回归测试通过
```

如果任务较小，也至少要说明怎么判断完成。

## 4. 单元测试

涉及以下内容时，应优先补充或更新单元测试：

* 核心函数
* 业务规则
* 数据转换
* 状态判断
* 模型输出解析
* 参数校验
* 错误处理

单元测试应覆盖：

* 正常输入
* 空输入
* 异常输入
* 边界值
* 状态变化
* 错误分支

禁止只测试成功路径。

## 5. 集成测试

涉及以下外部依赖时，应考虑集成测试或替代验证方式：

* 数据库
* 缓存
* 消息队列
* 对象存储
* 第三方接口
* 大模型服务
* OCR 服务
* 向量库
* 文件系统

集成测试应重点验证：

* 配置是否正确
* 外部服务是否可连接
* 超时和失败是否被处理
* 返回结构是否符合预期
* 数据是否正确落库或回滚

如果集成环境不可用，必须说明原因，并给出人工验证方式。

## 6. 回归测试

修改已有功能、接口、公共模块、配置、数据模型、权限逻辑时，必须考虑回归影响。

至少应确认：

* 原有核心功能是否仍可用
* 原有接口返回结构是否兼容
* 原有测试是否仍能通过
* 是否影响已有数据
* 是否影响部署或启动

禁止为了当前功能通过而破坏已有功能。

## 7. 接口测试

新增或修改接口时，必须验证：

* 正常请求
* 参数缺失
* 参数非法
* 未登录
* 无权限
* 资源不存在
* 状态不允许
* 外部服务失败
* 返回结构稳定性

接口返回必须符合项目既有格式。

不得随意改变已有接口字段含义。

## 8. 前端验收

涉及前端页面、交互、表单、列表、弹窗、状态展示时，必须检查：

* 加载状态
* 空数据状态
* 错误状态
* 提交中状态
* 重复点击
* 表单校验
* 接口失败提示
* 长任务进度反馈
* 移动端或小屏幕兼容性

禁止只验证有数据时页面正常。

## 9. AI 功能验收

涉及大模型、Agent、提示词、知识库、向量检索、智能生成时，必须额外验证：

* 输入过长时如何处理
* 模型返回空内容时如何处理
* 模型返回非法 JSON 时如何处理
* 模型返回格式不稳定时如何处理
* 模型调用超时时如何处理
* 模型服务切换后是否仍可用
* 生成结果是否经过必要校验
* 是否有成本、超时、重试或降级控制

禁止无校验地直接信任模型输出。

## 10. 异步任务验收

涉及后台任务、队列任务、定时任务、长时间任务时，必须验证：

* 任务是否能创建
* 任务状态是否正确流转
* 任务失败是否有记录
* 任务是否可以重试
* 重复执行是否幂等
* 超时是否能处理
* 用户是否能看到合理进度或结果

禁止只把任务提交到后台，而没有后续状态闭环。

## 11. 文件处理验收

涉及上传、下载、解析、导入、导出、临时文件时，必须验证：

* 文件类型限制
* 文件大小限制
* 文件名安全
* 路径安全
* 临时文件清理
* 失败后的残留处理
* 文件与数据库记录一致性
* 下载权限控制

禁止使用用户上传文件名直接拼接存储路径。

## 12. 验证命令规则

如果项目已有测试、检查、启动命令，必须优先使用项目现有命令。

常见验证命令包括但不限于：

```bash
pytest
pytest -q
go test ./...
npm test
npm run lint
npm run build
pnpm test
pnpm lint
pnpm build
docker compose up
```

不得随意编造项目不存在的命令。

执行验证后，必须记录：

* 执行了什么命令
* 是否通过
* 失败原因是什么
* 是否有警告
* 警告是否影响本次任务

如果命令无法执行，必须说明原因。

示例：

```text
未执行测试：当前环境缺少数据库服务。
未执行构建：项目未提供 build 脚本。
未执行集成测试：本地缺少外部模型服务配置。
```

禁止把“无法执行”包装成“已通过”。

## 13. 测试失败处理

如果测试失败，不得直接忽略。

必须先判断失败类型：

* 本次修改引入的问题
* 既有历史问题
* 环境缺失导致的问题
* 外部服务不可用导致的问题
* 测试用例本身过期或错误

如果是本次修改引入的问题，应优先修复。

如果是历史问题，应在总结中明确说明，不要伪装为本次已解决。

如果是环境问题，应说明缺失条件和建议的验证方式。

## 14. 警告处理

测试或构建过程中出现 warning 时，不能简单忽略。

必须判断 warning 是否影响本次修改。

以下 warning 应优先处理：

* 即将废弃的 API
* 测试标记未注册
* 配置项无效
* 依赖版本冲突
* 安全风险提示
* 类型错误或隐式转换
* 数据库迁移警告

如果暂不处理，应说明原因，并记录为后续建议。

## 15. 人工验收规则

不是所有任务都能完全自动化测试。

涉及以下内容时，可以使用人工验收：

* 页面效果
* 交互体验
* 生成内容质量
* 复杂业务流程
* 第三方系统联调
* 私有化部署环境

人工验收必须给出明确步骤：

1. 前置条件
2. 操作步骤
3. 预期结果
4. 实际结果
5. 是否通过

示例：

```text
人工验收：
1. 启动前后端服务
2. 登录测试账号
3. 创建一个目录生成任务
4. 等待任务完成
5. 查看生成目录是否返回结构化章节
结果：通过 / 未验证 / 失败
```

禁止只写“人工验证通过”而不说明验证过程。

## 16. 验收输出模板

任务完成后，必须按以下格式输出验证结果：

```text
验证结果：

1. 自动化测试
- 执行命令：
- 结果：
- 失败/警告：

2. 人工验证
- 验证场景：
- 验证步骤：
- 结果：

3. 回归影响
- 影响范围：
- 已确认：
- 未确认：

4. 风险与后续
- 风险：
- 后续建议：
```

如果某一项不适用，应写明“不涉及”，不要空着。

## 17. 提交前验收清单

准备提交代码前，必须完成自检：

* 是否有明确验收标准
* 是否执行了必要测试
* 是否记录了验证命令
* 是否说明了测试结果
* 是否处理了失败或警告
* 是否验证了错误分支
* 是否验证了回归影响
* 是否存在未验证的关键路径
* 是否有必要补充测试用例
* 是否有必要补充人工验收说明

如果存在未验证内容，必须在总结中明确说明。

## 18. 禁止事项

禁止以下行为：

* 未执行测试却声称测试通过
* 只验证成功路径，不验证失败路径
* 忽略测试失败
* 忽略关键 warning
* 编造不存在的验证命令
* 修改测试用例来掩盖真实问题
* 删除失败测试而不说明原因
* 为了通过测试降低业务校验
* 只写“已验证”，不说明验证方式
* 只完成功能开发，不提供验收证据
