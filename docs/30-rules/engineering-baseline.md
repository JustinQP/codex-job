# 工程基线

## 1. 目的

本文件定义项目开发、重构和交付必须遵守的工程底线。

项目是作者个人使用的 Codex 远程控制工具，不按多租户 SaaS 设计，但涉及本地代码目录、远程命令、Token、文件写入和长时间进程，仍必须保证基本可靠性和安全边界。

## 2. 总体原则

1. 优先解决真实使用问题，不为技术形式引入复杂架构。
2. 优先复用当前 FastAPI、SQLModel、SQLite、React、Vite、pytest 和 Codex 执行链路。
3. 允许必要重构，但重构必须服务于明确目标，并保持阶段可运行、可测试、可回退。
4. 不把临时 POC 约束无说明地带入正式主链路。
5. 不把个人工具过度设计为多租户、组织、RBAC、计费或高可用平台。
6. 不为了“以后可能需要”引入 Redis、Celery、Kafka、Kubernetes 或其他大型基础设施。

## 3. 配置

- URL、Token、数据目录、Runner/Agent ID、超时、并发数和路径白名单不得硬编码为生产值。
- 配置必须有明确默认值、解析规则和非法值处理。
- 示例配置不得包含真实 Token、账号或个人敏感路径。
- 新增配置时同步更新 README 或对应运行文档。
- Windows 是当前优先运行环境，路径、进程树和启动脚本必须考虑 Windows 行为。

## 4. 接口与领域边界

- Router 负责参数接收、认证和响应，不直接承载复杂状态机。
- Service 负责业务规则、状态流转和事务边界。
- 外部系统调用必须集中封装，不在多个模块重复拼接协议。
- 已发布接口不得无说明改变字段含义或状态语义。
- 错误响应应提供稳定的 HTTP 状态和可识别错误信息。
- 手机 API、Runner/Agent API 和内部 Bridge/进程接口应保持清晰边界。

## 5. 数据与迁移

- 数据库结构变更必须兼容已有 `app.db`。
- 不允许只修改 SQLModel 模型而忽略旧数据库升级。
- 迁移必须可重复检查，失败时不得伪装为成功。
- 破坏性变更前必须提供备份和回滚方式。
- 状态字段必须有集中定义和合法流转，不在各处直接写任意字符串。
- 幂等键、唯一约束和 lease 等可靠性规则应尽可能由数据库辅助保证。

## 6. 异步任务与进程

涉及 Run、Task、Session、Turn、Runner、Agent 或 App Server 时必须考虑：

- 创建是否幂等。
- 重复请求是否会重复执行。
- 状态是否有完整终态。
- 进程异常退出后如何恢复。
- 控制端或 Agent 重启后如何处理。
- 超时是否真的停止底层工作。
- 取消是否只改状态，还是实际终止进程。
- lease 是否续期、过期和回收。
- 同一目录是否存在并发写入冲突。
- 日志和事件是否支持增量、去重和断线补传。

禁止把任务简单提交到后台线程后就视为完成。

## 7. 本地目录与文件安全

- 设备执行端只能操作预先注册或白名单允许的工作目录。
- 控制端不得通过普通请求向设备下发任意绝对路径。
- 所有路径在使用前必须 `resolve` 并验证边界。
- Artifact 文件名和存储位置必须由服务端定义，不能直接使用用户输入拼接。
- 文件读取必须验证仍位于允许根目录。
- 日志、事件、请求体和 Artifact 必须有大小限制。
- API 响应和普通日志不应暴露不必要的绝对路径、Token 或环境变量。

## 8. 外部调用

- HTTP、Codex CLI、Codex App Server、Git 和子进程调用必须设置超时。
- 网络重试必须有限、有退避，并考虑请求幂等。
- 不对所有错误盲目重试。
- 外部响应必须校验类型和必要字段。
- 协议行为不确定时先增加验证脚本或测试，不基于猜测实现主链路。

## 9. 日志与可观测性

关键日志至少应能关联：

```text
request_id
device_id / runner_id
workspace_id / project_id
run_id / task_id
session_id / app_thread_id
turn_id / app_turn_id
command_id
```

要求：

- 使用明确日志级别。
- 失败日志包含处理阶段和可行动原因。
- 不记录 Token、完整敏感请求或无必要的大段模型输出。
- `print` 只允许用于非常早期脚本或最后兜底诊断；正式服务使用 logging。

## 10. 前端

- 优先复用当前组件、Hook、样式 token 和 API client。
- 不为简单状态引入大型状态管理库。
- 所有请求必须处理 loading、empty、error 和重复点击。
- 长任务必须显示运行状态、取消入口和恢复方式。
- 轮询不能无限重叠；实时流必须支持断开和页面卸载清理。
- 手机端必须明确显示当前设备、项目/Workspace 和执行模式，避免误操作。
- Token 存储和错误详情应符合当前可信网络定位，不误导为完整认证系统。

## 11. 测试

- 修改核心逻辑必须补充或更新测试。
- 状态机至少覆盖成功、失败、取消、超时、重复请求和非法状态。
- 数据库变更必须有迁移测试。
- 文件和路径逻辑必须有越界测试。
- 多设备调度必须验证命令不会被错误设备认领。
- 前端关键交互应逐步使用真实组件测试，不只依赖读取源码字符串。
- 真实 Codex smoke 与可在 CI 运行的 Fake/Mock 测试应分开。
- 测试要求和输出格式以 `testing-acceptance.md` 为准。

## 12. 依赖与重构

新增依赖前必须说明：

- 解决什么当前问题。
- 现有标准库或依赖为什么不能满足。
- 对安装、Windows、CI 和维护的影响。

重构要求：

- 先加回归保护，再移动核心代码。
- 每一步保持可启动和可验证。
- 保留必要兼容层，并明确删除条件。
- 不在同一任务中同时重写模型、接口、执行器和 UI，除非无法拆分且已说明风险。

## 13. 提交与交付

完成开发任务时必须说明：

```text
本次完成：
修改文件：
数据迁移：
验证命令与实际结果：
人工验证：
回归影响：
风险与未完成项：
回滚方式：
```

未经明确要求：

- 不自动 commit。
- 不 push。
- 不删除用户数据。
- 不修改真实项目目录中的无关文件。
- 不把未执行的测试描述为已通过。

## 14. 禁止事项

- 无认证暴露能够执行 Codex 或写入目录的接口。
- 直接信任客户端提供的设备、目录或文件路径。
- 使用 `shell=True` 拼接不可信输入。
- 无限制重试、无界队列、无界日志或无界事件缓存。
- 用修改测试掩盖真实业务缺陷。
- 在没有迁移方案时破坏已有 SQLite 数据。
- 为个人工具无实际需求地实现租户、组织、RBAC 或计费能力。
