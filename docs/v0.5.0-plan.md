
## v0.5.0：真正远程 Runner

### 目标

实现你最初想要的架构：

```text
云端 / 局域网后端
        ↑ HTTP
本机 Runner 主动轮询
        ↓
本地 Codex CLI + 本地项目目录
```

### 核心变化

Runner 不能再直接依赖：

```python
from backend.db import engine
```

而应该改成：

```text
runner client
  - BACKEND_URL
  - RUNNER_ID
  - RUNNER_TOKEN / API_TOKEN
```

### 新增 Runner HTTP API

建议增加一组专用接口：

```text
POST /runner/register
POST /runner/heartbeat
POST /runner/tasks/claim
POST /runner/tasks/{task_id}/log
POST /runner/tasks/{task_id}/finish
POST /runner/tasks/{task_id}/artifacts
GET  /runner/tasks/{task_id}/cancel-state
```

### 任务认领接口

不要让 Runner 直接 select SQLite。后端提供原子认领：

```text
POST /runner/tasks/claim

请求：
{
  "runner_id": "desktop-001"
}

响应：
{
  "task_id": 12,
  "project_path": "E:\\JustinQP\\xxx",
  "prompt": "...",
  "timeout_seconds": 7200,
  "task_type": "IMPLEMENT"
}
```

注意：这时路径是否返回给 Runner 要看部署方式。
如果后端也在本机或内网，返回路径可以接受；如果后端在公网云服务器，要考虑路径敏感信息。

### artifact 上传

真正远程后，后端不能再直接读 Runner 本地的：

```text
data/jobs/<task_id>/run.log
```

所以需要改成 Runner 执行后上传：

```text
run.log
result.md
diff.patch
git-status.txt
task-report.md
```

最简单做法：Runner 执行完成后，把这些文本通过 HTTP POST 回后端，后端保存到自己的 `data/jobs/<task_id>/`。

### v0.5.0 验收标准

```text
1. 后端和 Runner 不共享 SQLite 文件
2. 后端和 Runner 不共享 data 目录
3. Runner 通过 HTTP 注册
4. Runner 通过 HTTP 心跳
5. Runner 通过 HTTP claim 任务
6. Runner 通过 HTTP 回传日志、结果、diff
7. 后端页面能看到远程 Runner 回传的产物
```

这才算真正远程 Runner。
